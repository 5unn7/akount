New User Empty State Standardization

Akount Design System — Component specification & implementation plan

February 2026

1. The Principle

One pattern, three tiers, every domain.

Rule: A new user and an existing user see the same page layout. The only difference is:
• New users get setup action cards at the top (when setup is incomplete)
• All components render their natural empty states below (no skeletons, no fake previews)
• One shared EmptyState component handles all empty rendering (from @akount/ui)

No separate "empty page" components. No duplicated markup. The real components handle their own empty data.

2. EmptyState Component — Three Tiers

One component, three sizes. Used by every table, widget, and page across all domains.

Tier 1 — Inline (size="sm")

For widgets, sidebar items, and compact containers. Minimal — just an icon and one line of text.

No transactions yet
Use for: Dashboard widgets, sidebar summaries, card footers
Props: size="sm" variant="inline" icon={Receipt} title="No X yet"
Tier 2 — Card (size="md") — DEFAULT

For tables, lists, and content areas. Icon in a circle + title + description + optional action.

No invoices yet
Create your first invoice to start tracking revenue.
New Invoice
Use for: Table empty states, list pages, detail not found
Props: icon={FileText} title="No X yet" description="..."
Children: Optional action button(s) passed as children
Tier 3 — Hero (size="lg")

For domain landing pages and first-run experiences. Large icon with glow + prominent CTA.

No journal entries yet
Record your first transaction to start building your general ledger.
New Journal Entry Learn More
Use for: First-run domain pages, primary entity pages
Props: size="lg" icon={BookOpen} title="..." description="..."
Icon wrap: bg-primary/10 circle with breathing glow animation

Key behavior:
• Shown conditionally: {`{isNewUser && }`}
• Below it, the real dashboard components render with empty data (showing their natural empty states)
• Each card uses GlowCard with a color-matched glowColor
• First card gets btn-primary, rest get btn-outline
• Cards disappear automatically once setup conditions are met (no manual dismiss)
• Same pattern works for any domain: Banking, Invoicing, Planning, etc.

Don't: Create fake skeleton previews. Don't build separate empty page components. Let real components handle their own empty data.

6. Consolidation Plan

Steps to standardize all empty states across the app.

Step 1: Resolve the duplicate EmptyState
• packages/ui/src/patterns/EmptyState.tsx is the canonical source
• apps/web/src/components/shared/EmptyState.tsx has extra props: action, secondaryAction, progress
• Action: Merge the useful props (action, progress) into the canonical component, then delete the duplicate

Step 2: Standardize messaging
• Title pattern: "No [resources] yet" (friendly, not "found")
• Description pattern: "[Action verb] your first [resource] to [benefit]."
• Examples:
    "No invoices yet" / "Create your first invoice to start tracking revenue."
    "No transactions yet" / "Connect a bank account or import transactions to get started."
    "No journal entries yet" / "Record your first transaction to build your general ledger."

Step 3: Apply setup cards pattern to domains that need onboarding
• Banking: Setup cards for "Connect Bank" + "Add Manual Account"
• Overview: Migrate from custom EmptyState to @akount/ui + setup cards pattern
• Business: No setup cards needed (just better EmptyState CTAs)

Step 4: Audit remaining domains
• Check Planning, Insights, Services for empty state coverage
• Add EmptyState lg where missing
• Every list/table page should use EmptyState md from @akount/ui
Proposed canonical EmptyState props (merged)
interface EmptyStateProps {
  // Existing (keep)
  title: string;
  description?: string;
  icon?: LucideIcon;
  variant?: 'card' | 'inline';
  size?: 'sm' | 'md' | 'lg';
  children?: ReactNode;
  className?: string;

  // New (from app-specific version)
  action?: { label: string; href: string; };
  secondaryAction?: { label: string; href: string; };
}
Note: The progress prop from the app-specific version is only used in the overview page for onboarding progress. This is better handled by a separate OnboardingProgress component rather than bloating EmptyState. The action and secondaryAction props are convenience shortcuts — equivalent to passing <Button asChild><Link>...</Link></Button> as children, but cleaner for the common case.
